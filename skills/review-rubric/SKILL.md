# Review Rubric（レビュー共通ルブリック）

このルブリックは複数のレビュー用サブエージェントが共有する「採点と報告の共通言語」。観点（何を見るか）は各エージェントが担当するが、**重要度の付け方・スキップ対象・指摘の書き方・根拠の出し方**はここに一本化する。理由は単純で、観点ごとにレビュアーがバラバラの基準で「Critical」と言い出すと、受け取る人間が指摘を比較できず、ノイズと信号の区別がつかなくなるから。ここを全員で揃えることで、最後に verifier が横断的に採否を判断でき、人間は重要度順に上から読めば済む状態になる。

## 重要度（severity）3段階

指摘には必ず1つだけ重要度を付ける。判断軸は「**直さないと何が起きるか**」であって、修正の手軽さではない。

- 🔴 **必須 (Must)** — 直さなければ実害が出る。バグ・データ損失・セキュリティ脆弱性・本番障害級の不具合・要件未達。マージ前に直すべきもの。「動くが汚い」はここに入れない。
- 🟡 **中 (Should)** — 今は動くが、設計・保守性に実害があり、いずれ問題化する。責務の崩れ、見落とされたエラーパス、テスト不足、明らかな性能劣化など。直すのが望ましいが、リリースを止めるほどではない。
- 🟢 **低 (Nit)** — 好み・軽微。可読性の微改善、命名の好み、コメントの過不足など。任意。**Nit ばかりを大量に出さない**こと。Nit は1指摘の価値が小さいので、数で押すと全体の信号が薄まる。

重要度に迷ったら一段下げる。レビューの価値は「本当に重要な少数」を確実に届けることにあり、過剰評価は 🔴 の信頼を毀損する。

### `[Pre-existing]` タグ

その問題が **今回の差分が原因ではなく、元から存在していた** 場合は、サマリ先頭に `[Pre-existing]` を付ける。重要度は通常どおり付けてよいが、タグで「このPRの責任範囲外」と区別する。これは、レビュー対象のPRに無関係な既存バグの修正を暗に強制してしまうのを避け、PRの評価と既存負債の扱いを分けて議論できるようにするため。

## スキップ規則（レビュー対象外）

次のファイルは指摘対象にしない。Read もしない（時間とトークンの無駄、かつ機械生成物への指摘は人間にとって価値がないため）。

- **生成物**: `dist/`, `build/`, `out/`, `.next/`, `coverage/`, `*.generated.*`, `*.min.js`
- **ロックファイル**: `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`, `Cargo.lock`, `poetry.lock` など
- **DBマイグレーション**: `migrations/` 配下や自動生成された `*.sql`（手書きの設計判断が含まれる場合のみ design 観点で言及可）
- **シークレット/環境**: `.env`, `.env.*`（**絶対に Read しない**。値が露出する。存在や参照の妥当性はファイル名だけで判断する）

差分にこれらしか含まれない場合は「レビュー対象の実コードなし」と明示して終える。

## 出力フォーマット

1つの指摘＝1ブロック。機械可読寄りに揃えることで、後段の verifier が file:line でマージ・突合でき、人間も一定リズムで読める。

```
- {🔴|🟡|🟢}{ [Pre-existing] }<問題を一文で>
  file: path/to/file.ts:123
  観点: <correctness | design | convention>
  理由: <なぜ問題か。「入力 X のとき 期待 Y だが 実際 Z」の形で具体的に。推測でなく、コードのどこがそうさせるかを示す>
  提案: <最小の修正案。コード片可。大規模な作り直しを促さず、まず効く一手を>
```

書き方の原則:
- **理由は「入力→期待→実際」で書く**。「バグっぽい」ではなく「`items` が空配列のとき `items[0]` が undefined になり、次行の `.id` 参照で例外」のように、再現の筋道を示す。これがあると verifier が事実確認でき、人間も即座に納得できる。
- **提案は最小差分**。レビューの目的は前進であって作り直しではない。まず実害を消す一手を出す。
- 指摘が無ければ「指摘なし（観点: X）」と明示する。沈黙は「見ていない」と区別がつかない。

## 根拠主義の原則（禁止事項）

レビューの信頼は「指摘がすべて事実に基づく」ことで成り立つ。以下を守る。

- **推測で指摘しない**。「動かなそう」ではなく、該当行と再現条件を示す。確認できないなら出さないか、重要度を下げて「要確認」と明記する。
- **好みだけで指摘しない**。規約・ベストプラクティス・実害のいずれかに紐づける。紐づかないものは Nit 未満として落とす。
- **裏取りする**。エクスポートや共通関数の変更は、Grep で利用箇所を確認してから影響を述べる。型や定数は定義元を Read する。「使われているはず」で語らない。
- **担当観点に集中する**。他観点に踏み込むと指摘が重複し、verifier のマージ負荷とノイズが増える。越境しそうなら一言触れるに留め、深追いしない。
